# Implementation Roadmap To v0.7

This document is written for an implementation agent that needs explicit, low-ambiguity instructions. Follow it in order. Do not skip tests, documentation, release metadata, or GitHub release validation.

Current baseline: `v0.2.0`.

Primary project shape:

- CLI entry point: `src/GhActionsDoctor.Cli/Program.cs`
- CLI parsing and config loading: `src/GhActionsDoctor.Cli/Commands/`
- Core models: `src/GhActionsDoctor.Core/Models/`
- Workflow parser: `src/GhActionsDoctor.Core/Parsing/`
- Reporters: `src/GhActionsDoctor.Core/Reporting/`
- Rules: `src/GhActionsDoctor.Core/Rules/`
- Scanner: `src/GhActionsDoctor.Core/Scanning/`
- Tests: `tests/GhActionsDoctor.Tests/`
- Rule docs: `docs/rules/`
- Samples: `samples/good/` and `samples/bad/`

## Global Rules For Every Version

Use these rules for every task in this document.

1. Start each version from a clean `main`.
   ```bash
   git status --short
   git pull --ff-only origin main
   ```
2. Create a dedicated branch for the version.
   ```bash
   git switch -c release/v0.3.0
   ```
3. Keep commits small and reviewable. Prefer this order:
   - one commit for core models/rules,
   - one commit for CLI/reporters/action integration,
   - one commit for samples and tests,
   - one commit for documentation and version metadata.
4. Never commit generated output from `bin/`, `obj/`, `.tmp/`, or local tool installs.
5. For every new rule:
   - add one class under `src/GhActionsDoctor.Core/Rules/`,
   - register it in `src/GhActionsDoctor.Core/Rules/RuleCatalog.cs`,
   - add focused tests under `tests/GhActionsDoctor.Tests/Rules/`,
   - add or update a bad sample and, when useful, a good sample,
   - add one docs page under `docs/rules/`,
   - update `docs/rules/README.md`,
   - update the README rule table.
6. For every reporter:
   - add the format to `src/GhActionsDoctor.Core/Models/OutputFormat.cs`,
   - update CLI parsing in `ScanOptionsParser`,
   - update reporter selection in `Program.cs`,
   - add tests in `tests/GhActionsDoctor.Tests/Reporting/ReporterTests.cs`,
   - add CLI parser tests and at least one end-to-end scan test.
7. For every user-visible behavior change:
   - update README,
   - update `docs/configuration.md` if config is affected,
   - update `docs/project-status.md`,
   - update `CHANGELOG.md`.
8. Before each commit, run:
   ```bash
   dotnet build GhActionsDoctor.sln
   dotnet test GhActionsDoctor.sln
   git diff --check
   git status --short
   ```
9. Before each release, run:
   ```bash
   dotnet restore
   dotnet build GhActionsDoctor.sln --configuration Release
   dotnet test GhActionsDoctor.sln --configuration Release --no-build
   dotnet pack src/GhActionsDoctor.Cli --configuration Release --no-build
   dotnet run --project src/GhActionsDoctor.Cli -- scan --path samples/bad --fail-on none
   dotnet run --project src/GhActionsDoctor.Cli -- scan --path samples/bad --format json --fail-on none
   ```
10. Add any new format to release validation as soon as the format exists.
11. Local package smoke test before release:
   ```bash
   if (Test-Path .tmp/tools-vNEXT) { Remove-Item -LiteralPath .tmp/tools-vNEXT -Recurse -Force }
   dotnet tool install --tool-path .tmp/tools-vNEXT gh-actions-doctor --version NEXT_VERSION --add-source src/GhActionsDoctor.Cli/bin/Release
   .tmp/tools-vNEXT/gh-actions-doctor scan --path samples/bad --fail-on none
   ```
12. Release only after main CI is green.

## v0.3.0 - GitHub Actions Integration

Goal: make the tool easy to run inside GitHub Actions and produce native annotations.

### Task 1: Add GitHub Annotation Output

Output format: `github-annotations`

Files to touch:

- `src/GhActionsDoctor.Core/Models/OutputFormat.cs`
- `src/GhActionsDoctor.Core/Reporting/GitHubAnnotationsReporter.cs`
- `src/GhActionsDoctor.Cli/Commands/ScanOptionsParser.cs`
- `src/GhActionsDoctor.Cli/Program.cs`
- `tests/GhActionsDoctor.Tests/Reporting/ReporterTests.cs`
- `tests/GhActionsDoctor.Tests/Cli/ScanOptionsParserTests.cs`
- README CLI reference

Output behavior:

- Emit GitHub workflow commands:
  - `::error file=...,line=...,col=...,title=rule-id category::message Suggestion: ...`
  - `::warning ...`
  - `::notice ...` for info findings
- Escape workflow command special characters:
  - `%` to `%25`
  - carriage return to `%0D`
  - newline to `%0A`
  - `:` in properties to `%3A`
  - `,` in properties to `%2C`
- Include file path if present.
- Include line and column only when available.
- Do not print the normal text summary in this format.

Minimum tests:

- warning finding emits `::warning`,
- error finding emits `::error`,
- info finding emits `::notice`,
- file, line, and column are included,
- message escaping works for `%`, newline, comma, and colon,
- parser accepts `--format github-annotations`,
- invalid format message lists all supported formats.

Commit:


- YAML parse errors must be included as SARIF results under rule id `yaml-parse-error`.

Minimum tests:

- root version and `$schema` exist,
- rules include expected rule id and title,
- warning maps to `warning`,
- info maps to `note`,
- location uses normalized path,
- parse errors appear as SARIF results.

## v0.5.0 - Safe Auto-Fix

Goal: add a conservative `fix` command for changes that are mechanically safe. Do not attempt broad YAML rewriting.

### Task 1: Add Fix Command Skeleton

CLI command:

```text

```

Options:

```text
--path <path>
--rule <rule-id,...>
--dry-run
--config <path|none>
```

Files to touch:

- `src/GhActionsDoctor.Cli/Program.cs`
- new CLI command parser if needed under `src/GhActionsDoctor.Cli/Commands/`
- new fixer abstractions under `src/GhActionsDoctor.Core/Fixing/`
- tests under `tests/GhActionsDoctor.Tests/Cli/`

Behavior:

- Default is dry-run unless `--dry-run false` or an explicit apply flag is chosen. If the codebase prefers safer naming, use `--apply`.
- Print a clear list of files and fixes that would be applied.
- Exit code:
  - `0` if no fixable issues or fixes applied,
  - `1` if dry-run found fixable issues,
  - `2` for invalid usage,
  - `3` for parse errors.
- Do not modify files with parse errors.

Commit:

### Task 2: Implement Safe Fixes For Missing Timeouts

Rule: `missing-timeout`

Fix behavior:

- Add `timeout-minutes: 30` to jobs missing timeout.
- Preserve indentation.
- Insert near other job-level keys, preferably after `runs-on` if present.
- Do not change jobs that already have timeout.
- Do not modify generated YAML anchors or complex YAML nodes if the parser cannot safely identify the insertion point.

Minimum tests:

- dry-run reports intended insertion,
- apply adds `timeout-minutes: 30`,
- existing timeout is unchanged,
- invalid YAML is unchanged,
- two jobs in one file are both handled.

## v0.6.0 - Advanced Security Rules

Goal: improve security coverage while keeping findings explainable and low-noise.

### Task 1: Add OIDC Permission Risk Rule

Rule ID: `overbroad-id-token-permission`

Severity: `warning`

Category: `security`

Detection:

- Report `permissions: id-token: write` when no step appears to use cloud authentication or signing.
- Treat these as likely legitimate:
  - `aws-actions/configure-aws-credentials`
  - `google-github-actions/auth`
  - `azure/login`
  - `sigstore/cosign-installer`
  - run command containing `cosign sign`
- Do not report if `id-token: write` appears only in a job that uses one of the legitimate patterns.

Minimum tests:

- reports unused top-level `id-token: write`,
- does not report AWS auth,
- does not report Azure login,
- reports a job-level unused id-token permission.

Commit:

```bash
git add src tests docs README.md
git commit -m "feat: add overbroad id-token permission rule"
```

### Task 2: Add Untrusted Checkout Rule For pull_request_target

Rule ID: `pull-request-target-untrusted-checkout`

Severity: `error`

Category: `security`

Detection:

- Applies only to workflows triggered by `pull_request_target`.
- Report if a step checks out untrusted PR head code:
  - `actions/checkout` with `ref: ${{ github.event.pull_request.head.sha }}`
  - `ref: ${{ github.head_ref }}`
  - run commands that fetch or checkout PR head before privileged steps.
- Include job id and step name.
- This rule should be stricter than the existing `risky-pull-request-target` rule and may produce a separate finding.

Minimum tests:

- reports checkout of PR head SHA,
- reports checkout of head ref,
- does not report default checkout of base repository,
- does not run on normal `pull_request`.

Commit:

```bash
git add src tests docs README.md
git commit -m "feat: add untrusted checkout rule"
```

### Task 3: Add Shell Injection Pattern Rule

Rule ID: `untrusted-expression-in-run`

Severity: `warning`

Category: `security`

Detection:

- Report `run` commands that interpolate untrusted GitHub context values directly into shell.
- Initial risky expressions:
  - `${{ github.event.pull_request.title }}`
  - `${{ github.event.pull_request.body }}`
  - `${{ github.event.issue.title }}`
  - `${{ github.event.issue.body }}`
  - `${{ github.event.comment.body }}`
- Do not report when expression appears under `env:` and the shell command references the environment variable with quotes.
- Recommendation should suggest moving untrusted data to `env` and quoting it.

Minimum tests:

- reports PR title in run command,
- reports comment body in run command,
- does not report quoted env indirection,
- includes line and column when available.

Commit:

```bash
git add src tests docs README.md
git commit -m "feat: add untrusted expression rule"
```

## v0.7.0 - Baselines, Suppressions, And Adoption

Goal: make the tool easier to adopt in existing repositories without hiding new regressions.

### Task 1: Add Baseline File Support

Config field:

```yaml
baseline: .gh-actions-doctor-baseline.json
```

CLI option:

```text
--baseline <path|none>
```

Files to touch:

- `src/GhActionsDoctor.Core/Models/ScanOptions.cs`
- `src/GhActionsDoctor.Cli/Commands/ScanOptionsParser.cs`
- `src/GhActionsDoctor.Cli/Commands/ScanConfigLoader.cs`
- `src/GhActionsDoctor.Core/Scanning/WorkflowScanner.cs` or a new baseline filter service
- tests in CLI and scanner test files
- README and `docs/configuration.md`

Baseline matching key:

- rule id,
- normalized file path,
- job id when present,
- step name when present,
- stable message text.

Do not include line number in the key. Line numbers are unstable.

Baseline JSON:

```json
{
  "version": 1,
  "findings": [
    {
      "ruleId": "missing-timeout",
      "filePath": ".github/workflows/ci.yml",
      "jobId": "build",
      "stepName": "",
      "message": "Job 'build' does not define timeout-minutes."
    }
  ]
}
```

Minimum tests:

- matching finding is suppressed,
- non-matching finding remains,
- parse errors remain,
- missing baseline file returns usage error exit code `2`,
- config baseline works,
- `--baseline none` disables config baseline.

Commit:

```bash
git add src tests docs README.md
git commit -m "feat: add baseline suppression support"
```

### Task 3: Add Inline Suppression Comments

Supported syntax:

```yaml
# gh-actions-doctor-disable-next-line missing-timeout
# gh-actions-doctor-disable-file action-not-sha-pinned
```

Behavior:

- `disable-next-line` suppresses findings on the next non-empty line.
- `disable-file` suppresses all findings for that rule in the file.
- If a finding has no line, only file-level suppression can suppress it.
- Invalid rule ids are ignored.

Minimum tests:

- next-line suppression suppresses exactly one finding,
- file suppression suppresses all matching rule findings,
- unrelated rule remains,
- invalid rule id is ignored.

Commit:

```bash

git commit -m "feat: add inline suppression comments"
```
